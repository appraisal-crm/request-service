# ADR-008: Фронтенд единым монорепозиторием

> Русская версия · [English version](../../ADR-008-frontend-repo-structure.md)

**Статус:** Принято  
**Дата:** 2026-07-12

## Контекст

В системе четыре SPA на React + TypeScript — по одному на роль: клиент, оценщик,
инспектор, администратор. Это разные деплои с разной аудиторией, но у них большая
общая часть:

- **Авторизация** — один и тот же поток Keycloak OAuth2/OIDC, работа с токенами и
  guard'ы маршрутов.
- **Типы API-клиента** — DTO, сгенерированные из `swagger.json` каждого бэкенд-сервиса;
  все четыре приложения работают с одними и теми же контрактами request/inspect.
- **Дизайн-система** — общий UI-кит, темизация и i18n, чтобы приложения выглядели и
  вели себя единообразно.
- **Тулинг** — конфиги TypeScript, ESLint, Prettier и сборки.

Бэкенд — полирепо (каждый Go-сервис деплоится независимо — см.
[ADR-003](ADR-003-database-per-service.md)), и прежний чартер переносил это деление
на фронт как «четыре отдельных репозитория SPA». Для перечисленной общей части
четыре репо вынуждают либо дублировать код, либо вынести пятый «shared»-пакет в
реестр и поднимать его версию у четырёх потребителей — трение без выгоды для
маленькой команды.

## Решение

Фронтенд — **единый монорепозиторий**: `github.com/appraisal-crm/appraisal-frontend`,
на **pnpm workspaces + Turborepo**.

```
appraisal-frontend/
  apps/
    client/       appraiser/       inspector/       admin/     # 4 SPA, деплоятся независимо
  packages/
    ui/           # общая дизайн-система / библиотека компонентов
    api-client/   # типизированные API-клиенты из swagger.json сервисов
    auth/         # поток Keycloak OIDC, хранилище токенов, guard'ы маршрутов
    tsconfig/     eslint-config/                                # общий тулинг
```

Каждое приложение в `apps/` остаётся **независимо собираемым и деплоящимся**
артефактом — монорепо это выбор организации исходников и сборки, а не связанность
в рантайме. Общий код лежит в `packages/` и подключается через workspace, а не
через реестр.

Решение касается **только фронтенда**. Бэкенд остаётся полирепо;
[ADR-003](ADR-003-database-per-service.md) (база-на-сервис, без обращений к чужой
БД) не затрагивается — монорепо фронта не вносит рантайм-связанности между
бэкенд-сервисами.

## Последствия

**Положительные:**
- Общие авторизация, типы API и UI лежат в одном месте — без дублей и без плясок
  с версиями между репозиториями.
- Атомарные изменения по приложениям и общим пакетам едут одним PR/коммитом.
- Один тулчейн, конфиг lint/test/build и CI-пайплайн (Turborepo кэширует сборки).
- Изменение контракта в общем пакете сразу проверяется типами у всех четырёх
  потребителей.

**Отрицательные:**
- Репозиторий крупнее; контрибьютор клонирует все четыре приложения даже ради
  правки в одном.
- CI должен ограничивать работу затронутыми приложениями (affected-граф Turborepo),
  чтобы оставаться быстрым.
- Контроль доступа грубее, чем при отдельных репо (на уровне org, не на репо).
- Требует дисциплины монорепо (границы workspace, никаких кросс-импортов между
  приложениями мимо `packages/`).
