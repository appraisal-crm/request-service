# ADR-008: Фронтенд единым монорепозиторием

> Русская версия · [English version](../../ADR-008-frontend-repo-structure.md)

**Статус:** Принято  
**Дата:** 2026-07-12

## Контекст

Фронтенд — на React + TypeScript и обслуживает четыре роли: клиент, оценщик,
инспектор, администратор. У ролей большая общая часть:

- **Авторизация** — один и тот же поток Keycloak OAuth2/OIDC, работа с токенами и
  guard'ы маршрутов.
- **Типы API-клиента** — DTO, сгенерированные из `swagger.json` каждого сервиса;
  все приложения работают с одними контрактами request/inspect.
- **Дизайн-система** — общий UI-кит, темизация и i18n.
- **Тулинг** — конфиги TypeScript, ESLint, Prettier и сборки.

Надо было решить два вопроса: (1) один репозиторий или много и (2) сколько
деплой-приложений и по какой оси их делить.

Бэкенд — полирепо (каждый Go-сервис деплоится независимо — см.
[ADR-003](ADR-003-database-per-service.md)), и прежний чартер механически повторял
это как «четыре отдельных репозитория SPA, по одному на роль». Но роль — неверная
ось деления, а четыре репо вынуждают либо дублировать код, либо выносить пятый
«shared»-пакет в реестр — трение без выгоды для маленькой команды.

Значимая граница — **внешний / внутренний**, а не роль: `client` — публичный
портал (самообслуживание: подать и отслеживать заявку), а оценщик/инспектор/админ —
**внутренние инструменты сотрудников**, различающиеся лишь набором доступных
функций. Делить сотруднические инструменты по ролям смысла нет — одно приложение
показывает/скрывает функции по claim'у `realm_access.roles` из JWT. А вот держать
публичный портал клиента отдельно от сотруднического приложения стоит: в публичный
бандл не должны попадать внутренние UI/логика, и онбординг/авторизация у внешнего
клиента другие.

## Решение

Фронтенд — **единый монорепозиторий** (`github.com/appraisal-crm/appraisal-frontend`,
на **pnpm workspaces + Turborepo**), разбитый на два деплой-приложения по границе
внешний/внутренний:

```
frontend/
  apps/
    client/       # ВНЕШНИЙ публичный портал — request-service (подать / отслеживать заявку)
    office/       # ВНУТРЕННИЙ, гейт по роли внутри — request-service + inspect-service
                  #   оценщик:  статусы, назначение инспектора
                  #   инспектор: заполнение осмотра, фото, complete
                  #   админ:     добавим, когда появится его бэкенд
  packages/
    ui/           # общая дизайн-система / библиотека компонентов
    api-client/   # типизированные API-клиенты из swagger.json сервисов
    auth/         # Keycloak OIDC (Auth Code + PKCE), хранилище токенов, guard'ы по ролям
    tsconfig/     eslint-config/                                # общий тулинг
```

Внутри `office` роли разграничиваются по claim'у `realm_access.roles`, а не
отдельными приложениями. Каждое приложение в `apps/` собирается и деплоится
независимо; монорепо — это выбор организации исходников и сборки, а не связанность
в рантайме. Общий код лежит в `packages/` и подключается через workspace, а не
через реестр.

Решение касается **только фронтенда**. Бэкенд остаётся полирепо;
[ADR-003](ADR-003-database-per-service.md) не затрагивается — монорепо фронта не
вносит рантайм-связанности между бэкенд-сервисами.

## Последствия

**Положительные:**
- Общие авторизация, типы API и UI в одном месте — без дублей и плясок с версиями;
  атомарные изменения по приложениям и пакетам едут одним PR.
- Публичный бандл `client` не содержит внутренних UI и логики сотрудников.
- Новые роли сотрудников — это гейт внутри `office`, а не новое приложение/репо.
- Один тулчейн и CI-пайплайн; Turborepo кэширует и ограничивает сборку затронутыми
  приложениями.

**Отрицательные:**
- Репозиторий крупнее; контрибьютор клонирует оба приложения даже ради правки в одном.
- CI должен ограничивать работу затронутыми приложениями (affected-граф Turborepo).
- Контроль доступа грубее, чем при отдельных репо (на уровне org, не на репо).
- `office` обязан строго соблюдать гейт по ролям — баг там открывает функции между
  ролями внутри внутреннего приложения (внешний портал клиента при этом изолирован).
