# ADR-007: Надёжная публикация событий через Transactional Outbox

> Русская версия · [English version](../../ADR-007-event-delivery-outbox.md)

**Статус:** Принято  
**Дата:** 2026-07-10

Уточняет [ADR-001](ADR-001-kafka.md) — тот ADR выбрал Kafka как брокер; этот
решает, *как* события публикуются надёжно, как устроены топики и на какие
гарантии могут рассчитывать потребители.

## Контекст

При изменении доменного состояния (например, при смене статуса заявки) сервис
обязан сделать две вещи: сохранить изменение в PostgreSQL **и** опубликовать
событие в Kafka. Это две независимые системы без общей транзакции, поэтому
процесс может упасть между ними — **проблема двойной записи (dual-write)**:

- Сначала publish, потом commit → падение теряет запись в БД → событие сообщает
  о состоянии, которого нет (фантомное событие).
- Сначала commit, потом publish → падение теряет событие → состояние изменилось,
  но никого не уведомили и downstream-сервисы не отреагировали (потерянное событие).

Ни один порядок не безопасен, и никакие ретраи в хендлере это не чинят: две
разные системы нельзя записать атомарно. Также нужно зафиксировать раскладку
топиков, формат сообщений, семантику доставки и топологию брокера до того, как
писать продюсеров и консьюмеров, — потом менять это дорого.

## Решение

**1. Transactional Outbox для публикации.** Хендлеры никогда не пишут в Kafka
напрямую. Событие пишется в таблицу `outbox` в **той же транзакции БД**, что и
изменение состояния, — обе записи коммитятся атомарно или не коммитятся вовсе.
Фоновый relay-воркер (polling publisher) читает неопубликованные строки,
публикует их в Kafka и помечает отправленными. Недоступность Kafka лишь
задерживает доставку; ничего не теряется. Используем polling, а не CDC
(Debezium), чтобы не тащить лишнюю инфраструктуру.

**2. Один топик на сервис-продюсер.** События различаются по полю `event_type`,
а не топиком-на-событие:

| Топик            | Продюсер        | Типы событий                                 |
|------------------|-----------------|----------------------------------------------|
| `request.events` | request-service | `request.created`, `request.status_changed`  |
| `inspect.events` | inspect-service | `inspect.completed`                          |
| `review.events`  | review-service  | `report.ready`                               |

**Ключ сообщения** — идентификатор агрегата (например, `request_id`), поэтому все
события одного агрегата попадают в одну партицию и сохраняют порядок относительно
друг друга.

**3. JSON-envelope.** Без Schema Registry на MVP:

```json
{
  "event_id":    "uuid",
  "event_type":  "request.status_changed",
  "version":     1,
  "occurred_at": "2026-07-10T12:34:56Z",
  "request_id":  "uuid",
  "data":        { "old_status": "Appraisal", "new_status": "Report Sent", "...": "..." }
}
```

Эволюция схемы — только добавлением полей; ломающие изменения повышают `version`.

**4. Доставка at-least-once.** Консьюмер обрабатывает событие, затем коммитит
offset в Kafka. Вместе с ретраями outbox это означает, что дубли возможны,
поэтому каждый консьюмер ОБЯЗАН быть идемпотентным — дедуплицировать по
`event_id` (например, Redis `SET NX EX` или таблица с `ON CONFLICT DO NOTHING`).

**5. Режим KRaft, без Zookeeper.** Брокер работает в режиме KRaft — на один
компонент меньше в деплое и эксплуатации, и это поддерживаемая топология на
будущее.

**6. Клиентская библиотека: `segmentio/kafka-go`.** И relay-продюсер, и будущие
консьюмеры используют [`segmentio/kafka-go`](https://github.com/segmentio/kafka-go).
Это чистый Go с небольшим идиоматичным API `Writer`/`Reader`, который закрывает
наши нужды: polling-relay делает одну синхронную запись на строку outbox, а
consumer group напрямую ложится на `Reader`. Альтернатива, `franz-go`, быстрее и
богаче (транзакции, exactly-once, свой пул соединений), но тяжелее и рассчитана
на пропускную способность, которой у нас на MVP нет. Выбор `kafka-go` держит
зависимости минимальными, в духе принципа проекта «no magic frameworks».
Вернуться к вопросу, если понадобятся транзакции Kafka или упрёмся в пропускную
способность, которую простой `Writer` не тянет.

## Последствия

**Положительные:**
- Нет потерянных и фантомных событий — состояние в БД и строка outbox атомарны.
- Новые потребители (аналитика, аудит) подписываются без правок продюсеров.
- Порядок per-агрегат гарантирован ключом по идентификатору агрегата.
- Минимум инфраструктуры — нет Zookeeper, Schema Registry, CDC-пайплайна.

**Отрицательные:**
- Таблицу `outbox` и relay-воркер нужно написать, запускать и мониторить в каждом
  сервисе-продюсере.
- At-least-once означает, что дубли — норма, а не исключение; идемпотентность
  обязательна в каждом консьюмере, а не опциональна.
- Polling-relay добавляет задержку в пределах интервала опроса (смягчается
  коротким интервалом; при текущем объёме приемлемо).
- JSON крупнее и менее строг, чем Avro/Protobuf; переход на Schema Registry
  отложен до тех пор, пока этого не потребует объём или строгость контракта.
